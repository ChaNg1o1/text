# 产品需求说明

## 目标

本文档从内部产品需求角度描述 `text` 当前应覆盖的能力范围，用于指导研发、维护和后续需求讨论。内容基于现有实现收敛，不作为全新功能立项说明。

## 核心功能需求

### 1. 数据导入

系统需要支持将多源文本和相关上下文装配为统一分析请求。

需求点：

- 支持 `.txt`、`.csv`、`.json`、`.jsonl` 与目录导入
- 支持从上传结果中解析 `texts`
- 支持附带 `artifacts`、`activity_events`、`interaction_edges`
- 支持记录案件级元数据 `case_metadata`

成功标准：

- 非结构化输入可转换为统一 `AnalysisRequest`
- 导入失败时给出明确错误，而不是静默忽略

### 2. 任务配置

系统需要允许用户在分析前选择任务类型、LLM 后端和任务参数。

当前应支持的任务类型：

- 综合分析 `full`
- 作者验证 `verification`
- 闭集识别 `closed_set_id`
- 开集识别 `open_set_id`
- 聚类 `clustering`
- 画像 `profiling`
- 傀儡检测 `sockpuppet`

当前应支持的参数类型：

- 可疑文本 ID
- 参考作者 ID
- 候选作者 ID
- 聚类文本 ID
- 画像对象 ID
- 账号 ID
- `top_k`

### 3. 特征提取

系统需要在可接受时延内完成确定性特征提取，并优先复用缓存结果。

需求点：

- 支持批量抽取
- 支持关闭缓存
- 支持仅提取特征、不进行 LLM 分析
- 依赖缺失时允许优雅降级

### 4. 多智能体分析

系统需要将同一份输入送入多个学科视角并行分析，而不是依赖单一模型输出。

需求点：

- 每个学科 Agent 独立产出结构化报告
- Agent 异常不能导致整个请求直接崩溃
- 最终必须有综合 Agent 给出收敛性结论

### 5. 结果查看与管理

系统需要提供结果查看、历史管理与过程追踪能力。

需求点：

- 查看分析列表
- 查看单次分析详情
- 查看进度流
- 查看特征页
- 删除分析
- 取消运行中分析

### 6. 配置管理

系统需要允许内部成员管理后端模型和运行参数。

需求点：

- 维护 LLM backend 配置
- 查看健康状态与可观测数据
- 调整部分运行时设置

## 输入数据要求

### 文本

每条文本至少需要：

- `id`
- `author`
- `content`

可选字段：

- `timestamp`
- `source`
- `metadata`
- `artifact_id`
- `content_sha256`
- `derivation_kind`

### 活动与关系

在 `sockpuppet` 等任务中，除文本以外还应允许引入：

- 活动事件时间线
- 账号交互边
- 关联元数据

## 输出需求

系统输出至少应覆盖：

- 结构化报告对象
- 每个 Agent 的摘要和发现
- 综合结论等级
- 证据条目
- 运行状态与时间信息
- 可序列化导出格式

## 非功能需求

### 1. 可解释性

结论不能只有“像”或“不像”，必须尽量给出：

- 证据维度
- 置信度
- 异常样本
- 多学科交叉验证结果

### 2. 可维护性

系统需保持模块边界清晰，至少区分：

- 数据摄取
- 特征抽取
- Agent 推理
- 综合判断
- API 交互
- 前端展示

### 3. 可扩展性

后续应能较低成本扩展：

- 新任务类型
- 新 Agent
- 新 LLM backend
- 新前端视图

### 4. 稳定性

系统在以下场景下应尽量优雅退化：

- Rust 扩展缺失
- NLP 模型未安装
- 某个 Agent 调用失败
- 后端 sidecar 启动失败

## 当前已知约束

- LLM backend 依赖外部配置和密钥注入
- API 与桌面端运行路径不同，桌面端通过 sidecar 提供后端
- 文档和历史 `README` 可能存在命令偏差，维护时应以源码入口为准
